Configure CDC Connector Parameters

Set up Source, Destination, and Ingestion parameters for the Openflow connector.

After the connector is deployed, configure three parameter groups by right-clicking the PostgreSQL process group on the canvas and selecting Parameters.

Step 8.1: Upload JDBC Driver

The JDBC driver must be uploaded before applying parameters. It is attached directly inside the Source Parameters:

  1. Download the PostgreSQL JDBC driver from jdbc.postgresql.org (version 42.7.x or later)
  2. Right-click the PostgreSQL process group → Parameters → Source Parameters
  3. Find the PostgreSQL JDBC Driver parameter
  4. Check the Reference asset checkbox next to the field
  5. Upload the postgresql-42.7.x.jar file using the file picker that appears
ⓘ
No separate Assets menu

There is no standalone Assets menu. The JDBC driver is attached directly to the PostgreSQL JDBC Driver parameter by enabling the Reference asset checkbox within the parameter field itself.

Step 8.2: Source Parameters (PostgreSQL Connection)

ParameterValue
PostgreSQL Connection URLjdbc:postgresql://YOUR_INSTANCE_HOST.YOUR-ACCOUNT.YOUR-REGION.aws.postgres.snowflake.app:5432/postgres
PostgreSQL Usernamesnowflake_admin
PostgreSQL PasswordYour Postgres instance password
Publication Nameecommerce_cdc_publication
⚠
JDBC URL format

The URL must start with jdbc:postgresql:// (not postgresql://). Get the hostname from SHOW POSTGRES INSTANCES; in Snowflake. Ensure no trailing spaces.

Step 8.3: Destination Parameters (Snowflake)

ParameterValue
Destination DatabaseECOMMERCE_CDC
Destination Schema Pattern${source.schema.name}
Snowflake RoleOPENFLOW_RUNTIME_ROLE
Snowflake WarehouseOPENFLOW_WH
Snowflake Authentication StrategySNOWFLAKE_MANAGED_TOKEN
⚠
Critical: all four destination fields are required
  • Destination Schema Pattern is required — without it, the connector has nowhere to write schemas and no data will appear in Snowflake
  • Snowflake Role must be OPENFLOW_RUNTIME_ROLE, not ACCOUNTADMIN — ACCOUNTADMIN is explicitly blocked by Openflow and causes a "role name is invalid" error
  • Snowflake Authentication Strategy must be SNOWFLAKE_MANAGED_TOKEN for Snowflake Deployments (SPCS)

Step 8.4: Ingestion Parameters

ParameterValueDescription
Ingestion TypefullFull snapshot first, then incremental CDC. Required — do not remove.
Included Table Namesecommerce.customers, ecommerce.products, ecommerce.ordersExplicit list of tables to replicate (schema.table format)
Merge Task Schedule CRON* * * * * ?Continuous merge so data appears in Snowflake immediately
ⓘ
Why data may not appear in Snowflake immediately

The connector uses a two-phase process: data is first streamed into journal tables (e.g., customers_JOURNAL_xxx_1), then the merge task (triggered by the CRON) merges it into the final destination tables. Using * * * * * ? triggers continuous merging so data appears within seconds. After the merge, query tables in ECOMMERCE_CDC."ecommerce" using double-quoted identifiers: SELECT * FROM ECOMMERCE_CDC."ecommerce"."customers";

Step 8.5: Apply and Start

  1. Click Apply to save all parameter configurations
  2. Right-click on the canvas (outside the process group) and select Enable all Controller Services
  3. Wait until every service shows ENABLED status before proceeding
  4. Right-click the PostgreSQL process group → Start
✅ Expected Healthy State

A healthy running connector shows:

  • Snapshot Load: all processors running, Read/write shows KB of data, 0 warnings, 0 invalid
  • Incremental Load: all processors running, 0 invalid
  • Metrics and Stream Staleness Prevention: running
  • No bulletin errors

Step 8.6: Confirm Initial Load Complete

To confirm the snapshot has finished and the connector switched to incremental mode:

  1. Right-click PostgreSQL process group → Controller Services
  2. Find Table State Store → click ⋮ → View State
  3. All 3 tables should show INCREMENTAL_REPLICATION
StateMeaning
NEWTable scheduled but snapshot not started yet
SNAPSHOT_REPLICATIONInitial snapshot in progress
INCREMENTAL_REPLICATIONSnapshot complete, streaming live changes ✅
FAILEDPermanent failure — see troubleshooting below

Troubleshooting

FAILED, MISSING_PRIMARY_KEYS in Table State Store

Caused by setting REPLICA IDENTITY FULL on tables that have primary keys. The connector requires REPLICA IDENTITY DEFAULT for primary key tables.

Fix in PostgreSQL:

SQL (PostgreSQL) 
ALTER TABLE ecommerce.customers REPLICA IDENTITY DEFAULT;
ALTER TABLE ecommerce.products REPLICA IDENTITY DEFAULT;
ALTER TABLE ecommerce.orders REPLICA IDENTITY DEFAULT;

Then in Openflow: Stop the connector → Controller Services → Table State Store → Clear state → Start.

✖1 Invalid Processor in Stream Staleness Prevention

The deprecated Fetch Journal Streams processor (in the yellow "Deprecated, drain only mode" box) becomes invalid because its PostgreSQL controller service was never configured. Since it has nothing queued, safely disable it:

  1. Double-click PostgreSQL → double-click Stream Staleness Prevention
  2. Right-click Fetch Journal Streams → Disable
  3. Return to parent canvas and restart the connector

Connector running but 0 bytes / no data in Snowflake

Check in order:

  1. Ingestion Parameters: confirm Ingestion Type = full and Included Table Names are set — removing either causes silent 0-byte operation
  2. Destination Schema Pattern: must be set to ${source.schema.name}
  3. Table State Store: check for FAILED tables and clear state if found
  4. Controller Services: ensure all show ENABLED before starting

Restart after any configuration change

  1. Right-click process group → Stop
  2. Right-click process group → Parameters → fix values → Apply
  3. Right-click canvas → Enable all Controller Services
  4. Right-click process group → Start
