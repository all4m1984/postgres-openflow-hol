Appendix: Openflow & NiFi Glossary

Key terms and concepts used throughout this lab, explained for first-time Openflow and NiFi users.

Openflow Terms NiFi / Canvas Terms Parameter Groups CDC & Replication Terms UI Symbols & Indicators Table Replication States

Openflow Terms

Openflow Openflow

Snowflake's managed data integration product built on Apache NiFi. It lets you move data between external sources (databases, SaaS apps, streams) and Snowflake without writing custom code. Openflow manages the infrastructure; you configure connectors through a visual UI.

Deployment Openflow

The control plane of Openflow. A Deployment is the top-level Openflow infrastructure that you create once per account. It manages authentication, routing, and the overall environment. Think of it as the "server" that hosts your runtimes. In this lab we use a Snowflake Deployment, which runs on Snowpark Container Services (SPCS) inside your Snowflake account. The alternative is BYOC (Bring Your Own Cloud), which runs in your own AWS account.

Runtime Openflow

The execution environment within a Deployment. A Runtime is a containerized NiFi instance (running on SPCS) where connectors and flows actually execute. Each Runtime has a size (Small, Medium, Large), a dedicated Snowflake Role, and External Access Integrations defining which external hosts it can reach. You can have multiple Runtimes per Deployment — one per connector is the recommended practice.

Connector Openflow

A pre-built, Snowflake-supported data pipeline for a specific source (e.g., PostgreSQL, MySQL, Salesforce). A connector is packaged as a NiFi flow and deployed into a Runtime from the Openflow connectors catalog. Once deployed, it appears on the canvas as a process group that you configure via Parameters.

Data Plane Integration Openflow

A Snowflake account-level object that authorizes Openflow to create and manage infrastructure (compute pools, container services) in your account. Created automatically when you create a Snowflake Deployment. Requires the CREATE OPENFLOW DATA PLANE INTEGRATION privilege.

Runtime Integration Openflow

A Snowflake account-level object representing a specific Runtime instance. Each Runtime you create corresponds to one Runtime Integration. Requires the CREATE OPENFLOW RUNTIME INTEGRATION privilege.

External Access Integration (EAI) Openflow

A Snowflake object that grants an SPCS Runtime permission to make outbound network connections to specific external hosts and ports. Because SPCS containers are network-isolated by default, every external connection (e.g., to your PostgreSQL instance) must be explicitly allowed through an EAI backed by a Network Rule. Without an EAI, the connector cannot reach the source database.

Runtime Role Openflow

The Snowflake role assigned to a Runtime. All Snowflake operations performed by the connector (writing tables, creating schemas, using warehouses) run under this role. It must have access to the destination database, warehouse, and any EAIs used by the Runtime. ACCOUNTADMIN cannot be used as a Runtime Role — Openflow explicitly blocks it.

NiFi / Canvas Terms

Canvas NiFi

The visual workspace you see when you open a Runtime in the Openflow UI. The canvas is where flows, processors, and connections are displayed and arranged. You interact with flows by right-clicking on the canvas or on individual components. It is provided by Apache NiFi, which is the underlying engine Openflow is built on.

Flow NiFi

A data pipeline defined on the canvas. A flow consists of Processors connected by relationships (arrows). Data enters as FlowFiles, gets transformed or routed by Processors, and exits through output connections. In Openflow, each connector installs as a pre-built flow.

FlowFile NiFi

The unit of data in NiFi. A FlowFile is a record that moves through the pipeline. It has two parts: attributes (metadata key-value pairs, like table name or record count) and content (the actual data payload, e.g., a batch of rows). The "In / Read/write / Out" metrics on a processor or process group count FlowFiles passing through.

Processor NiFi

The fundamental building block of a NiFi flow. Each Processor performs a specific task: reading from a database, transforming data, routing based on conditions, or writing to Snowflake. Processors have configurable Properties (settings) and output Relationships (success, failure, etc.). Examples in the PostgreSQL connector: CaptureChangePostgreSQL, PutSnowpipeStreaming, ExecuteSQL.

Process Group NiFi

A container that groups related Processors together on the canvas. Process groups help organize complex flows. Double-clicking a process group enters it and shows its internal processors. In the PostgreSQL connector, the top-level group is "PostgreSQL", which contains sub-groups: Snapshot Load, Incremental Load, Metrics, and Stream Staleness Prevention. You right-click a process group to Start, Stop, or access Parameters.

Connection / Queue NiFi

The arrows between Processors on the canvas. Each connection carries FlowFiles of a specific relationship (e.g., "success", "failure"). Connections act as queues — FlowFiles wait in the queue until the downstream Processor is ready to consume them. The "Queued" metric shows how many FlowFiles are waiting. A healthy running connector typically shows 0 queued.

Controller Services NiFi

Shared services that multiple Processors can use. Common examples: database connection pools (DBCPConnectionPool), Snowpipe Streaming clients, and the Table State Store. Controller Services must be ENABLED before the processors that depend on them can run. Access them by right-clicking the canvas → "Controller Services" or by right-clicking a process group → "Controller Services". If a Controller Service is not enabled, all dependent processors will fail to initialize.

Bulletin NiFi

An error or warning message generated by a Processor. Bulletins appear as small notification icons on a processor and in the Bulletin Board. They are the primary way to diagnose connector errors. A healthy connector shows 0 bulletins. Always check bulletins first when troubleshooting — right-click a processor → "View Bulletins", or check the process group's bulletin count in the stats bar.

Parameter Context NiFi

A named set of key-value configuration variables that Processors reference using the #{parameter_name} syntax. In the PostgreSQL connector there are three contexts: Source Parameters, Destination Parameters, and Ingestion Parameters. Contexts are bound to a process group — right-click the process group → "Parameters" to view and edit them. Changes take effect after clicking Apply; the connector does not need to be reinstalled.

Relationship NiFi

The named output path of a Processor. When a Processor finishes handling a FlowFile, it sends it down a specific relationship (e.g., "success" or "failure"). Each relationship connects to a downstream Processor or loops back. Unhandled relationships cause FlowFiles to pile up in queues.

Parameter Groups (PostgreSQL Connector)

Source Parameters Openflow

Configuration for the PostgreSQL source database. Defines how the connector connects to the source: JDBC connection URL, username, password, publication name, JDBC driver file, and replication slot name. These tell the connector where to read data from.

Destination Parameters Openflow

Configuration for the Snowflake destination. Defines where replicated data is written: Destination Database, Destination Schema Pattern, Snowflake Role, Warehouse, and Authentication Strategy. The Destination Schema Pattern is required — it determines which schema inside the destination database the connector creates. Using ${source.schema.name} mirrors the source schema name.

Ingestion Parameters Openflow

Configuration for which tables to replicate and how. Key parameters: Ingestion Type (full = snapshot first then incremental; incremental = skip snapshot), Included Table Names or Included Table Regex (which tables to replicate), and Merge Task Schedule CRON (how often journal data is merged into destination tables). Without Ingestion Type or Included Table Names, the connector runs silently with 0 bytes transferred.

CDC & Replication Terms

CDC (Change Data Capture) CDC

A technique for tracking and capturing row-level changes (INSERT, UPDATE, DELETE) in a source database and streaming them to a target. Instead of copying the entire table periodically, CDC reads only the changes, making replication continuous and near real-time.

Logical Replication / WAL CDC

PostgreSQL's mechanism for CDC. The Write-Ahead Log (WAL) records every change made to the database. Logical replication decodes WAL entries into a human-readable change stream. The Openflow connector reads this stream via a replication slot. Requires wal_level = logical in PostgreSQL configuration.

Publication CDC

A PostgreSQL object that defines which tables are included in logical replication. The Openflow connector reads only tables listed in the publication. Created with CREATE PUBLICATION. A connector cannot replicate a table unless it is included in the publication configured in Source Parameters.

Replication Slot CDC

A PostgreSQL mechanism that ensures WAL data is retained until the connector has consumed it. The Openflow connector creates its own replication slot automatically. Important: if the connector is paused or removed, the slot remains and PostgreSQL keeps accumulating WAL — this can cause disk space issues on high-traffic databases. Drop the slot manually if you permanently remove the connector.

Snapshot Load CDC

The first phase of replication when Ingestion Type = full. The connector reads the current state of all configured tables in bulk and writes them to Snowflake. A table is in SNAPSHOT_REPLICATION state during this phase. Once complete, the table transitions to INCREMENTAL_REPLICATION. Visible as the "Snapshot Load" process group on the canvas.

Incremental Load CDC

The ongoing CDC phase after the snapshot completes. The connector continuously reads new changes (INSERT, UPDATE, DELETE) from the PostgreSQL WAL and streams them to Snowflake. Visible as the "Incremental Load" process group on the canvas.

Journal Table CDC

An intermediate staging table created by the connector in Snowflake (e.g., customers_JOURNAL_1234567890_1). Incoming changes are first written here via Snowpipe Streaming. The merge task then reads the journal and applies changes to the final destination table. If you see journal tables but no destination table, the merge CRON has not yet run.

Soft Delete / _SNOWFLAKE_DELETED CDC

When a row is deleted in PostgreSQL, the connector does not physically remove it from the Snowflake destination table. Instead, it sets the metadata column _SNOWFLAKE_DELETED = TRUE. This preserves the full history of changes. To query only active (non-deleted) rows, always filter with WHERE "_SNOWFLAKE_DELETED" = FALSE. Other metadata columns added to every row: _SNOWFLAKE_INSERTED_AT and _SNOWFLAKE_UPDATED_AT.

REPLICA IDENTITY CDC

A PostgreSQL table setting that controls which columns are included in WAL entries for UPDATE and DELETE operations. The Openflow connector requires REPLICA IDENTITY DEFAULT for tables with primary keys (this is PostgreSQL's default). Do not set it to FULL for tables with primary keys — this causes the connector to report MISSING_PRIMARY_KEYS and permanently fail replication for that table.

UI Symbols & Indicators

Each process group and processor on the canvas shows a stats bar with colored icons. Here is what each means:

SymbolNameMeaning
▶RunningNumber of processors currently running (active). A healthy connector shows all processors as running.
■StoppedNumber of processors that are stopped. All stopped = connector is paused. Some stopped = partial run, usually indicates a problem.
▲InvalidNumber of processors with a configuration error (missing required property, disabled controller service). Even one invalid processor can prevent the whole flow from working correctly. Fix before starting.
⌨DisabledNumber of processors intentionally disabled. Disabled processors do not run and do not generate errors.
✓ValidNumber of processors with valid configuration. Shown at the bottom of the stats bar.
✱Stopped (alt)Alternate stopped indicator shown in some NiFi versions.
●BulletinA blue dot on a processor means it has a recent bulletin (error/warning message). Click or right-click → View Bulletins to read the message.
InInFlowFiles received by this group in the last 5 minutes. 0 (0 bytes) → 0 means nothing has flowed in — usually indicates misconfigured Ingestion Parameters.
Read/writeRead/writeBytes read from and written to external systems (databases, Snowflake). Non-zero values confirm data is actively flowing.
OutOutFlowFiles sent out by this group. A value like 1 → 3 means 1 FlowFile in the last interval, 3 total.
QueuedQueuedFlowFiles currently waiting in connections (queues) to be processed. Should normally be 0. A growing queue indicates a slow or blocked downstream processor.

Table Replication States (Table State Store)

The Table State Store controller service tracks the replication state of each table. View it via: right-click process group → Controller Services → Table State Store → ⋮ → View State.

StateMeaningAction
NEWTable is queued for replication but snapshot has not started yet.Wait — it will start automatically.
SNAPSHOT_REPLICATIONInitial bulk copy of the table is in progress.Wait — will transition to INCREMENTAL when done.
INCREMENTAL_REPLICATIONSnapshot complete. Connector is streaming live changes. ✅Normal operating state. No action needed.
FAILEDReplication permanently stopped due to an unrecoverable error (e.g., MISSING_PRIMARY_KEYS).Fix the root cause in PostgreSQL, clear the Table State Store, then restart the connector.
