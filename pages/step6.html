Set Up Openflow Deployment & Runtime

Create the Openflow infrastructure (deployment + runtime) that will host our CDC connector.

Before deploying a CDC connector, you need an Openflow Deployment (the control plane) and a Runtime (the execution environment). This step sets up both using the Snowflake-managed deployment option (SPCS).

Step 6.1: Configure Core Snowflake Privileges

SQL (Snowflake) 
-- Grant Openflow admin privileges
USE ROLE ACCOUNTADMIN;

-- Create a dedicated Openflow admin role
CREATE ROLE IF NOT EXISTS OPENFLOW_ADMIN;
GRANT ROLE OPENFLOW_ADMIN TO ROLE ACCOUNTADMIN;

-- Grant required Openflow privileges
GRANT CREATE OPENFLOW DATA PLANE INTEGRATION ON ACCOUNT TO ROLE OPENFLOW_ADMIN;
GRANT CREATE OPENFLOW RUNTIME INTEGRATION ON ACCOUNT TO ROLE OPENFLOW_ADMIN;
GRANT CREATE COMPUTE POOL ON ACCOUNT TO ROLE OPENFLOW_ADMIN;

-- Set default role to OPENFLOW_ADMIN (ACCOUNTADMIN is blocked from Openflow runtimes)
SET current_user_name = CURRENT_USER();
ALTER USER IDENTIFIER($current_user_name) SET DEFAULT_ROLE = 'OPENFLOW_ADMIN';

-- Set secondary roles for your user (required for Openflow auth)
ALTER USER IDENTIFIER($current_user_name) SET DEFAULT_SECONDARY_ROLES = ('ALL');

-- Grant Openflow admin role to your user
GRANT ROLE OPENFLOW_ADMIN TO USER IDENTIFIER($current_user_name);
⚠
ACCOUNTADMIN cannot access Openflow runtimes

Per Snowflake documentation, users with a default role of ACCOUNTADMIN are explicitly blocked from logging into Openflow - Snowflake Deployment runtimes. You must set your default role to a different role (e.g., OPENFLOW_ADMIN) before accessing the Openflow UI. If you skip this step, you will see the error: "The role requested has been explicitly blocked for use with this application by an administrator."

ⓘ
Why set DEFAULT_SECONDARY_ROLES?

Snowflake sessions have a primary role (the active role) and optional secondary roles that provide additional privileges. By setting DEFAULT_SECONDARY_ROLES = ('ALL'), your user automatically activates all granted roles (including OPENFLOW_ADMIN) as secondary roles in every session. This is required because the Openflow control plane authenticates using your user context and needs access to the OPENFLOW_ADMIN privileges—even when your primary role is set to something else (e.g., SYSADMIN). Without this setting, the Openflow UI may fail to authorize actions because the required role is not active in your session.

✅ Expected Result

All statements execute successfully. The OPENFLOW_ADMIN role is created with the required privileges.

Step 6.2: Create the Openflow Deployment

  1. In Snowsight, navigate to Ingestion → Openflow
  2. Click Launch Openflow (a new tab opens)
  3. Click Create a deployment
  4. In the Prerequisites step, confirm all requirements are met → click Next
  5. In the Deployment location step:
    • Select Snowflake as the deployment location
    • Enter a deployment name: openflow_lab_deployment
  6. Click Create Deployment
✅ Expected Result

The deployment is created and visible in the Deployments tab with status Active.

ⓘ
Snowflake Deployment vs BYOC

We are using Snowflake Deployment (runs on Snowpark Container Services) which is simpler to set up. The alternative is BYOC (Bring Your Own Cloud) which runs in your own AWS account via CloudFormation.

Step 6.3: Create the Snowflake Role for the Runtime

The runtime needs a dedicated Snowflake role for accessing resources:

ⓘ
Why a dedicated role and warehouse?

Although Openflow runs on SPCS (compute containers), it still needs to interact with Snowflake objects—writing CDC data into tables, creating schemas, and managing ingestion. These Snowflake operations require a role (to authorize access to databases, schemas, and tables) and a virtual warehouse (to provide compute for certain Snowflake operations like table creation and schema management). Think of it this way: SPCS handles the CDC connector execution (reading from Postgres, transforming data), while the role and warehouse handle the Snowflake-side authorization and compute. A dedicated role also follows the principle of least privilege—the runtime only has access to the specific resources it needs, keeping your account secure.

SQL (Snowflake) 
USE ROLE ACCOUNTADMIN;

-- Create a dedicated role for the Openflow runtime
CREATE ROLE IF NOT EXISTS OPENFLOW_RUNTIME_ROLE;

-- Grant the role to the Openflow admin
GRANT ROLE OPENFLOW_RUNTIME_ROLE TO ROLE OPENFLOW_ADMIN;
GRANT ROLE OPENFLOW_RUNTIME_ROLE TO ROLE ACCOUNTADMIN;

-- Create a warehouse for Openflow processing
CREATE WAREHOUSE IF NOT EXISTS OPENFLOW_WH
  WAREHOUSE_SIZE = 'XSMALL'
  AUTO_SUSPEND = 60
  AUTO_RESUME = TRUE;

-- Grant warehouse access to the runtime role
GRANT USAGE, OPERATE ON WAREHOUSE OPENFLOW_WH TO ROLE OPENFLOW_RUNTIME_ROLE;

-- Create the destination database for CDC data
CREATE DATABASE IF NOT EXISTS ECOMMERCE_CDC;

-- Grant database access to the runtime role
GRANT USAGE ON DATABASE ECOMMERCE_CDC TO ROLE OPENFLOW_RUNTIME_ROLE;
GRANT CREATE SCHEMA ON DATABASE ECOMMERCE_CDC TO ROLE OPENFLOW_RUNTIME_ROLE;

-- Grant role to your user
GRANT ROLE OPENFLOW_RUNTIME_ROLE TO USER IDENTIFIER($current_user_name);
✅ Expected Result

All statements execute successfully. The runtime role has access to the warehouse and destination database.

Step 6.4: Create an External Access Integration (EAI)

The runtime needs network access to connect to your Snowflake Postgres instance:

ⓘ
Why is an External Access Integration (EAI) required?

Openflow Runtimes running on Snowpark Container Services (SPCS) are network-isolated by default—all outbound (egress) traffic is blocked. This is a security measure to prevent containers from making unauthorized connections. To allow the CDC connector to reach your Postgres instance, you must explicitly whitelist the destination host and port through a Network Rule, then wrap it in an External Access Integration. The EAI is then attached to the runtime, granting it permission to connect to only the specified endpoints. This ensures a zero-trust network posture where each runtime can only access the exact external services it needs.

SQL (Snowflake) 
USE ROLE ACCOUNTADMIN;

-- Create a network rule allowing access to Snowflake Postgres
-- Replace the host with your actual Postgres instance hostname
CREATE OR REPLACE NETWORK RULE openflow_postgres_network_rule
  MODE = EGRESS
  TYPE = HOST_PORT
  VALUE_LIST = ('YOUR_INSTANCE_HOST.YOUR-ACCOUNT.YOUR-REGION.aws.postgres.snowflake.app:5432');

-- Create the External Access Integration
CREATE OR REPLACE EXTERNAL ACCESS INTEGRATION openflow_postgres_eai
  ALLOWED_NETWORK_RULES = (openflow_postgres_network_rule)
  ENABLED = TRUE;

-- Grant the EAI to the runtime role
GRANT USAGE ON INTEGRATION openflow_postgres_eai TO ROLE OPENFLOW_RUNTIME_ROLE;
⚠
Replace the hostname!

Replace YOUR_INSTANCE_HOST.YOUR-ACCOUNT.YOUR-REGION.aws.postgres.snowflake.app with the actual hostname from your Postgres instance (Step 1). You can find it in the output of SHOW POSTGRES INSTANCES;

✅ Expected Result

Network rule and EAI are created. The runtime role has usage permission on the integration.

Step 6.5: Create the Runtime

  1. Return to the Openflow Control Plane tab in your browser
  2. Click Create a runtime
  3. Configure the runtime:
    FieldValue
    Runtime Nameopenflow_lab_runtime
    Deploymentopenflow_lab_deployment
    Node TypeSmall
    Min/Max Nodes1 / 1
    Snowflake RoleOPENFLOW_RUNTIME_ROLE
    External Access Integrationsopenflow_postgres_eai
  4. Click Create
ⓘ
Runtime Creation Time

The runtime takes 2-5 minutes to start. Wait for it to show Active status in the Runtimes tab.

✅ Expected Result

The runtime appears in the Runtimes tab with status Active. Clicking on it opens the Openflow canvas (empty at this point).

Step 6.6: Configure Allowed Domains

After the runtime is created, ensure the PostgreSQL JDBC connector domain is allowed. Navigate to the runtime settings and verify that your Postgres hostname is in the allowed list (it should be covered by the EAI you created above).

✔
Openflow Infrastructure Ready!

Your Openflow deployment and runtime are active. Next, we'll deploy the PostgreSQL CDC connector into this runtime.
